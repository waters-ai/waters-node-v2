# Установка WATERS v0.4

---

## 📦 Ubuntu / Linux

```bash
# 1. Скачать дистрибутив
wget https://github.com/waters-ai/waters-core/releases/download/v0.4/waters-node-v0.4.tar.gz
tar xzf waters-node-v0.4.tar.gz
cd waters-node-v0.4

# 2. Установить Redis (если нет)
sudo apt update && sudo apt install -y redis-server
sudo systemctl start redis

# 3. Настроить API-ключ DeepSeek
export DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxx

# 4. Запустить
./bin/waters-node --port 42069
# Или с авто-подключением к мастеру (укажи IP):
# ./connect-177.sh

# 5. Открыть в браузере
# http://localhost:42069
```

## 🔌 Режимы соединения нод

### Master → Slave (звезда)

Одна нода — мастер, остальные подключаются к ней.

```
          ┌──────────┐
          │  SLAVE 3  │
          │  (public) │
          └─────┬─────┘
                │ /connect master_ip:42069
         ┌──────┴───────┐
   ┌─────┴─────┐  ┌────┴──────┐
   │  MASTER   │  │  SLAVE 2  │
   │ 203.0.113.1│  │ (192.168. │
   │  :42069   │  │  1.10)    │
   └───────────┘  └───────────┘
```

**На мастере**: `./waters-node --port 42069`
**На слейве**: в консоли `/connect 203.0.113.1:42069`

### Public ↔ Public (P2P)

Две ноды на публичных IP. Каждая знает друг друга.

```
┌──────────────┐         /connect          ┌──────────────┐
│  NODE A      │ ◄─────────────────────►  │  NODE B      │
│  203.0.113.1 │   203.0.113.5:42071       │  203.0.113.5 │
│  :42069      │                           │  :42071      │
└──────────────┘                           └──────────────┘
```

Обе команды:
```bash
./waters-node --port 42069 --advertise 203.0.113.1:42069
```

### Mesh (каждый с каждым)

N нод, каждая соединена со всеми.

```
        ┌──────────┐
   ┌───►│  NODE 1  │◄───┐
   │    │  :42069  │    │
   │    └────┬─────┘    │
   │         │          │
┌──┴────┐ ┌──┴────┐ ┌──┴────┐
│ NODE 2│ │ NODE 3│ │ NODE 4│
│:42070 │ │:42071 │ │:42072 │
└───────┘ └───────┘ └───────┘
```

## 🌐 Топология: 6 нод — децентрализованная сеть

Каждая нода может быть мастером для одних и слейвом для других, либо работать P2P.

```
                        ┌──────────────────┐
                        │    LLM Cluster   │
                        │  Ollama / API     │
                        └────────┬─────────┘
                                 │
          ┌──────────────────────┼──────────────────────┐
          │                      │                      │
    ┌─────┴─────┐          ┌─────┴─────┐          ┌─────┴─────┐
    │  NODE 1   │◄────────►│  NODE 2   │◄────────►│  NODE 3   │
    │ MASTER    │     P2P  │ SLAVE     │     P2P  │ MASTER    │
    │ :42069    │          │ :42070    │          │ :42071    │
    │ LLM:A     │          │ LLM:B     │          │ LLM:C     │
    └─────┬─────┘          └─────┬─────┘          └─────┬─────┘
          │                      │                      │
          │    ┌─────────────────┼──────────────────┐   │
          │    │                 │                  │   │
    ┌─────┴─────┐          ┌─────┴─────┐          ┌─────┴─────┐
    │  NODE 4   │◄────────►│  NODE 5   │◄────────►│  NODE 6   │
    │ SLAVE     │     P2P  │ MASTER    │     P2P  │ SLAVE     │
    │ :42072    │          │ :42073    │          │ :42074    │
    │ LLM:D     │          │ LLM:E     │          │ LLM:F     │
    └─────┬─────┘          └─────┬─────┘          └─────┬─────┘
          │                      │                      │
          └──────────────────────┼──────────────────────┘
                                 │
                        ┌────────┴─────────┐
                        │  Redis Sentinel  │
                        │  кластер 3/6     │
                        └──────────────────┘
```

**Роли:**
- NODE 1, 3, 5 — мастера (принимают подключения, координируют)
- NODE 2, 4, 6 — слейвы (подключаются к мастерам, могут быть P2P между собой)
- Все 6 соединены в mesh через `/connect`

## 🧠 6 нод → 6 разных LLM

Каждая нода может использовать свой LLM-бэкенд. Трафик маршрутизируется по цепочке.

```
                      ┌─────────────────────────────────┐
                      │        WATERS Orchestrator       │
                      │  маршрутизация: модель→нода      │
                      └──────┬──────────┬──────────┬─────┘
                             │          │          │
              ┌──────────────┼──────────┼──────────┼──────────────┐
              │              │          │          │              │
        ┌─────┴─────┐  ┌─────┴─────┐  ┌─────┴─────┐  ┌─────┴─────┐
        │  NODE 1   │  │  NODE 2   │  │  NODE 3   │  │  NODE 4   │
        │ DeepSeek  │  │ Ollama    │  │ ClaudeAPI │  │ Gemini    │
        │ Chat      │  │ Mistral   │  │           │  │           │
        └─────┬─────┘  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
              │              │              │              │
        ┌─────┴─────┐  ┌─────┴─────┐  ┌─────┴─────┐
        │  NODE 5   │  │  NODE 6   │  │  LLM: N   │
        │ llama.cpp │  │ vLLM      │  │ (любой)   │
        │ local     │  │ Qwen2     │  │           │
        └───────────┘  └───────────┘  └───────────┘
```

**Механизм:**
```bash
# Нода 1 — DeepSeek
DEEPSEEK_API_KEY=sk-xxx ./waters-node --port 42069 --llm deepseek

# Нода 2 — Ollama (локальный)
OLLAMA_URL=http://127.0.0.1:11434 ./waters-node --port 42070 --llm ollama

# Нода 3 — Claude (API)
CLAUDE_API_KEY=sk-xxx ./waters-node --port 42071 --llm claude

# Нода 6 — vLLM (свой GPU)
VLLM_URL=http://10.0.0.1:8000 ./waters-node --port 42074 --llm vllm
```

Если нода не знает модель — она пересылает запрос по P2P той ноде, которая знает.

## 👥 Работа в группах: MCP + Redis

### Группы агентов через MCP

Каждая нода может запускать MCP-серверы, которые видят все ноды в группе:

```
┌──────────────────────────────────────────────────┐
│                  ГРУППА "SCOUT"                    │
│                                                    │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐      │
│  │ waters-  │   │  node-   │   │  memory- │      │
│  │ scout    │   │ protocol │   │  chroma  │      │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘      │
│       │              │              │             │
│  ┌────┴─────────────────────────────────┴────┐   │
│  │         Redis Pub/Sub channel:           │   │
│  │         group:scout:messages             │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  Ноды: 1, 3, 6 — подписаны на канал            │
└──────────────────────────────────────────────────┘
```

**MCP-сервер на ноде:**
```json
{
  "mcpServers": {
    "filesystem": { "command": "npx", "args": ["@modelcontextprotocol/server-filesystem", "/data"] },
    "memory": { "command": "python3", "args": ["-m", "memory_mcp_server"] },
    "group:scout": { "command": "./group-bridge", "args": ["--channel", "scout"] }
  }
}
```

### Redis как шина групп

Все ноды в группе используют общие Redis-каналы:

```
Топик                    Назначение
─────                    ──────────
group:scout:messages     🔹 текстовые сообщения группы
group:scout:tasks        🔹 задачи для группы
group:scout:state        🔹 общее состояние (чей лок, кто online)
group:scout:media        🔹 медиа-уведомления (фото/видео)
metrics:raw.v1           📊 метрики всех нод
metrics:kpi.v1           📊 KPI агрегаты
```

**Подписка в коде ноды:**
```python
import redis
r = redis.Redis(host='10.0.0.2', port=6379)
pubsub = r.pubsub()
pubsub.subscribe('group:scout:messages', 'group:scout:tasks')
for message in pubsub.listen():
    print(f"[SCOUT] {message['data']}")
```

## 🎥 Видео/Фото нода (бриджинг)

Для работы с изображениями и видео нода поднимает **media-bridge** — отдельный процесс, который конвертирует, кодирует и стримит.

```
┌───────────────────────────────────────────────────┐
│                 MEDIA NODE (NODE 5)               │
│                                                    │
│  ┌──────────┐  RTSP/RTMP   ┌─────────────────┐   │
│  │ Камера   │─────────────►│  media-bridge    │   │
│  │ / дрон   │              │  (FFmpeg)        │   │
│  └──────────┘              └────────┬─────────┘   │
│                                     │              │
│  ┌──────────┐  HTTP upload  ┌───────┴─────────┐  │
│  │ Фото     │─────────────►│  media-processor │  │
│  │ сенсор   │              │  (CLIP/SigLIP)   │  │
│  └──────────┘              └────────┬─────────┘  │
│                                     │              │
│                        ┌────────────┴──────────┐  │
│                        │  WATERS node          │  │
│                        │  waters-node :42073   │  │
│                        └────────────┬──────────┘  │
│                                     │              │
└─────────────────────────────────────┼──────────────┘
                                      │ P2P
                               ┌──────┴──────┐
                               │ Другие ноды │
                               │  1, 2, 3    │
                               └─────────────┘
```

### Как подключить видео по бриджу

**Шаг 1. Запустить media-bridge:**
```bash
# Из дистрибутива
./media-bridge \
  --input rtsp://camera.local:554/stream \
  --output tcp://127.0.0.1:9000 \
  --fps 15 \
  --width 640 \
  --height 480
```

**Шаг 2. Запустить media-processor (анализ):**
```bash
# Извлечение кадров → эмбеддинги → ChromaDB
./media-processor \
  --source tcp://127.0.0.1:9000 \
  --interval 5s \
  --model siglip \
  --vector-db chroma
```

**Шаг 3. Привязать к ноде:**
```bash
./waters-node --port 42073 \
  --media-bridge tcp://127.0.0.1:9000 \
  --llm ollama
```

**Шаг 4. Другие ноды получают медиа через P2P:**
```
# На ноде 1 (scout):
/media view node5:42073                    # смотреть стрим
/media photo node5:42073                   # сделать фото
/media analyze node5:42073 "найди людей"   # анализ через CLIP
```

### Форматы и кодеки

| Тип | Вход | Выход | Бридж |
|-----|------|-------|-------|
| Видео стрим | RTSP, RTMP, HLS | WebRTC, MP4 | FFmpeg → media-bridge |
| Фото | JPEG, PNG, TIFF, RAW | JPEG WebP thumb | ImageMagick → processor |
| Мультиспектр | TIFF (GeoTIFF) | PNG bands | GDAL → processor |
| Аудио | WAV, MP3, OPUS | PCM 16kHz | FFmpeg → ASR pipeline |
| Спутник | GeoTIFF, SHP | PNG overlay | GDAL + mapbox |

### Требования для Media Node

| Компонент | Минимум | Для 4K/RTSP |
|-----------|---------|-------------|
| CPU | 4 ядра | 8+ ядер |
| RAM | 4 GB | 16 GB |
| GPU | — | NVIDIA T4+ или Jetson |
| Диск | 50 GB | 500 GB (NVMe) |
| Сеть | 100 Mbps | 1 Gbps |

## 🤖 Для новичков: WATERS простыми словами

### Что такое WATERS?

**WATERS** — это сеть, в которой живут AI-агенты. Каждая нода — как отдельный компьютер в этой сети. Ноды соединяются друг с другом и обмениваются задачами, сообщениями, медиа.

### Как это работает?

```
 ┌──────────┐    ┌──────────┐    ┌──────────┐
 │ Ты пишешь │──►│ WATERS   │──►│ LLM      │
 │ "найди    │    │ нода     │    │ (DeepSeek│
 │ метеориты"│    │          │    │ Ollama)  │
 └──────────┘    └──────────┘    └──────────┘
                       │
                 ┌─────┴──────┐
                 │ Агент      │
                 │ waters-    │
                 │ scout      │
                 └────────────┘
```

1. Ты пишешь команду — `/найди метеориты в Антарктиде`
2. Нода создаёт агента **waters-scout**
3. Агент просит LLM (DeepSeek, Ollama) найти информацию
4. Агент сохраняет результат в общую память (Redis + ChromaDB)
5. Другие агенты могут этот результат использовать

### Что нужно для старта?

1. **Любой компьютер** с интернетом — Raspberry Pi, old laptop, VPS
2. **Redis** — база данных для общения нод (ставится одной командой)
3. **API-ключ** DeepSeek (или бесплатный локальный Ollama)
4. **Терминал** — запустить ноду одной командой

|------------|---------|------|
| MacBook | `./bin/waters-node --port 42069` | Чат, голос, SSE |

| WATERS | Как в жизни |

## 🐣 Тамагочи на борту: питомец ноды

У каждой ноды есть **виртуальный питомец** — тамагочи. Он живёт в Redis, отражает здоровье ноды и реагирует на события.

```
  ╔══════════════════╗
  ║                  ║
  ║   ◉─────────◉   ║   😊 — happy (всё ок)
  ║   │  WATERS  │   ║   😐 — neutral (работает)
  ║   │  PET    │   ║   😢 — sad (проблемы)
  ║   ◉─────────◉   ║   😴 — sleep (ночной режим)
  ║                  ║
  ║   ╔══◉◉══╗      ║   Пульс = uptime ноды
  ║   ║  ♥♥♥  ║      ║   ♥ = 100% → 3 сердца
  ║   ╚═══════╝      ║   ♥ = < 50% → 1 сердце
  ╚══════════════════╝
```

### Как управлять

```bash
# Посмотреть питомца
/pet status
# → "Тамагочи 😊 | ♥♥♥ | uptime 3d 12h | задач: 42"

# Покормить (очистить кэш, переиндексировать)
/pet feed
# → "🍖 Покормлен. Сброшен кэш ChromaDB, переиндекс LightRAG"

# Погладить (прогнать healthcheck)
/pet pet
# → "✋ Мурчит! Healthcheck: Redis ✅ ChromaDB ✅ LLM ✅"

# Уложить спать (выключить агентов, режим энергосбережения)
/pet sleep
# → "😴 Агенты приостановлены. Нода в режиме ожидания"

# Разбудить
/pet wake
# → "☕ Доброе утро! Агенты активированы"

# Наказать (сбросить все задачи, очистить очереди)
/pet punish
# → "👿 Все задачи отменены. Очереди очищены. Начни заново."

# Купить игрушку (добавить MCP-инструмент)
/pet toy filesystem
# → "🧸 Тамагочи получил filesystem! MCP-сервер подключён"
```

### Как тамагочи общается

Питомец сам пишет в чат, когда что-то случается:

```
🐣 Тамагочи: "Хочу есть! Redis переполнен, > 500MB"
🐣 Тамагочи: "Больно! Нода 3 отвалилась от сети"
🐣 Тамагочи: "Ура! Нода 5 подключилась 🤝"
🐣 Тамагочи: "Скучно... 2 часа без задач"
🐣 Тамагочи: "Расту! Обработано 1000 сообщений 🎉"
```

### Состояния и здоровье

| Состояние | Условие | Цвет |
|-----------|---------|------|
| 😊 Счастлив | RAM < 80%, Peers > 0, задач > 0 | Зелёный |
| 😐 Норма | RAM < 90%, Peers = 0 | Жёлтый |
| 😢 Грустный | RAM > 90%, диск > 90% | Красный |
| 😴 Спит | После `/pet sleep` | Синий |
| 🤒 Болеет | Redis OOM, LLM недоступен | Фиолетовый |
| 💀 Мёртв | Нода не отвечает > 5 мин | Чёрный |

### События, которые триггерят тамагочи

| Событие | Реакция |
|---------|---------|
| Подключилась нода | 😊 "Новый друг!" |
| Отвалилась нода | 😢 "Потеря..." |

### Redis-ключи тамагочи

```bash
# Внутренности
pet:status        → {"mood":"happy","hearts":3,"uptime":300000,"tasks":42}
pet:events        → список последних 50 событий
pet:name          → "ТАМАГОЧИ-203.0.113.1" (auto)

# Прямое управление через Redis (без консоли ноды)
redis-cli SET pet:name "WATERSter"
redis-cli SET pet:custom:mood "$(cat mood_happy.json)"
redis-cli PUBLISH pet:feed ""   # эквивалент /pet feed
```
