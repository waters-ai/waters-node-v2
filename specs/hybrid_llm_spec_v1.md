# Техническое задание: Гибридная LLM-система waters-node v0.6

**Версия:** 1.0
**Дата:** 2026-05-18
**Автор:** Конструктор Сети (`agent.constructor.v1`)
**Статус:** Проектирование
**Назначение:** Дополнение к архитектурной спецификации waters-node

---

## 1. Проблема

Распределённая сеть waters-node состоит из 2-6 устройств с RAM от 2 до 32 ГБ.
Канал связи с DeepSeek (облако/ЦОД) **нестабилен**:
- Нода может быть на связи от 1 до 15 минут
- Период отсутствия связи — от 15 минут до нескольких часов
- Заранее предсказать появление/пропадание канала невозможно

**Ключевое ограничение:** полная LLM (DeepSeek-R1) не помещается ни на одной ноде.
Максимум — 32 ГБ, что достаточно для Q4-квантованной 7B модели.

**Требуется:** максимальное качество решения задач на каждой ноде при частичном или полном отсутствии связи с DeepSeek, используя асинхронный вклад всех устройств.

---

## 2. Цель

Создать гибридную систему, где:

| Условие | Режим | Качество |
|---------|-------|----------|
| Есть связь с DeepSeek | Полный режим: DeepSeek + все ноды как распределённые вычислители | ~100% |
| Связь нестабильна (15 мин/15 мин) | DTN-фрагментация: ЦОД готовит ситуации → ноды выполняют → асинхронный кворум | 75-90% от DeepSeek |
| Связи нет часами | Полная автономия: L0-L1 на борту + локальная ассоциативная память | 60-75% |
| Нода 2 ГБ | Только L0: Марковские цепи / АВО / SDM | 30-40% как вспомогательная нода |
| Нода 32 ГБ | L0+L1+L2: Tensor Train + фрагменты DeepSeek | 85-90% |

## 3. Научная база (методы)

### 3.1. Американская школа

| Метод | Автор/год | Применение | Модуль |
|-------|-----------|------------|--------|
| **Sparse Distributed Memory (SDM)** | Kanerva (1988) | Ассоциативное восстановление контекста из 10% совпадений. 4 ГБ → 10⁹ векторов | `src/sdm.rs` |
| **Compressed Sensing** | Candès–Tao–Donoho (2004) | Редкая проекция знаний DeepSeek → восстановление из 10% данных | `src/sensing.rs` |
| **Product Quantization (PQ)** | Jégou et al (2011) | Векторный поиск: кодовая книга → 8 байт вместо 768. Поиск за 50 ns | `src/pq.rs` |
| **Speculative Decoding** | Stern et al (2018) | L0 генерирует гипотезу → DeepSeek верифицирует при появлении канала | `src/spec_decode.rs` |
| **Differentiable Neural Computer (DNC)** | Graves et al (2016) | 6 нод = 6 страниц внешней памяти одной машины | `src/dnc.rs` |
| **Federated Split Learning** | McMahan (2017) | Нода считает embedding → ЦОД классифицирует | Встроено в `src/situator.rs` |
| **Bayesian Committee Machines (BCM)** | Tresp (1993) | N частичных ответов → байесово взвешивание в ЦОД | `src/federator.rs` |

### 3.2. Японская школа

| Метод | Автор/год | Применение | Модуль |
|-------|-----------|------------|--------|
| **FGCS (Parallel Inference)** | ICOT (1982-1992) | Логический скелет задачи → параллельный вывод на всех нодах | `src/logic.rs` |
| **Amari Associative Memory** | Amari (1972) | Неполный контекст → полная ситуация через ортогональные проекции | `src/assoc_mem.rs` |
| **Neocognitron (Fukushima)** | Fukushima (1980) | Послойное выделение признаков без учителя — L0 | `src/neocognitron.rs` |

### 3.3. Советская школа

| Метод | Автор/год | Применение | Модуль |
|-------|-----------|------------|--------|
| **Тензорные поезда (TT)** | Оселедец–Тыртышников (2009) | Сжатие DeepSeek-слоя 500 МБ → 5 МБ без потерь. Ключ для 8-16 ГБ | `src/tensor.rs` |
| **Автоматы Цетлина** | Цетлин (1963) | Автомат в случайной среде: когда слать/ждать/работать локально | `src/tsetin.rs` |
| **Ситуационное управление** | Поспелов (1970-80) | ГРАМ — граф альтернатив и метапланов. Ядро фрагментации | `src/situator.rs` |
| **АВО (кворум)** | Журавлёв (1978) | Алгебра вычисления оценок: голосование с весами | `src/federator.rs` |
| **МГУА** | Ивахненко (1965) | Самоорганизация моделей: L1-модель растёт от фрагментов DeepSeek | `src/mgua.rs` |
| **Адаптивные ДП** | Айзерман (1970) | Обучение с подкреплением в нестабильной среде | `src/aizer.rs` |

### 4.2. DTN-протокол обмена

При появлении канала:

```
OUTGOING (нода → ЦОД):
  ┌─ Results Queue — с last_seq → ответы на ситуации
  └─ Health/state — сколько памяти, какие модули активны

INCOMING (ЦОД → нода):
  ┌─ Situations Pack — {skeleton, rules, depends_on, weight}
  │  1-5 КБ каждая, 500-5000 за 15 мин окна
  ├─ Context Pack — похожие решения из ChromaDB (SDM/PQ)
  │  20-200 КБ, 50-300 шт за 15 мин
  ├─ L2 Weight Update — обновление весов МГУА
  │  0.1-1 КБ, 1000+ шт
  └─ Tensor Train Slice — новая порция сжатой модели
      200-500 КБ, 10-50 шт
```

### 4.3. Алгоритм работы ноды (автомат Цетлина)

```
Состояния:
  S0 — ОЖИДАНИЕ (нет связи)        → работа на L0-L1
  S1 — ПРИЁМ (канал открыт)        → priority: сдать results → принять ситуации
  S2 — ВЫПОЛНЕНИЕ (есть контекст)  → Situator: {скелет + правила} → L0/L1/L2
  S3 — ОТВЕТ (результат готов)     → в очередь на отправку
  
Переходы:
  S0 → S1 : обнаружен канал
  S1 → S2 : пакет принят, есть работа
  S1 → S0 : канал потерян
  S2 → S3 : ситуация решена
  S3 → S0 : → СБРОС/результат ждёт
  S2 → S0 : работа прервана (→ offline queue)
```

## 5. Спецификация модулей

### 5.1. `src/situator.rs` — Ситуационное ядро (Поспелов) ~600 строк

**Ядро всей системы.** Отвечает за фрагментацию/сборку задач.

```
struct Situation {
    id: SID,                  // "s_0047"
    task_id: TID,              // ID родительской задачи
    depends_on: Vec<SID>,      // зависимости от других ситуаций
    context_hash: [u8; 32],   // blake3 хеш контекста
    rules: Vec<Rule>,          // продукционные правила
    skeleton: String,          // prompt-шаблон
    weight: f64,               // важность для итога
    schema: OutputSchema,      // что ожидаем на выходе
    ttl: Duration,             // время жизни ситуации
}

struct Situator {
    graph: DagMap<SID, Situation>,  // граф ситуаций (поспеловский ГРАМ)
    queue_out: VecDeque<SituPack>,  // что передать ноде
    queue_in: VecDeque<SituResult>, // что получено от ноды
}
```

Методы:
- `fn fragment_task(task_str: &str) → Vec<Situation>` — разбить задачу на ситуации
- `fn pack_for_dtn(situations: &[SID]) → Vec<u8>` — упаковать ситуации для DTN
- `fn unpack_from_dtn(data: &[u8]) → Vec<Situation>` — распаковать на ноде
- `fn aggregate(results: &[SituResult]) → TaskResult` — собрать ответы кворумом

### 5.2. `src/sdm.rs` — Sparse Distributed Memory (Kanerva) ~400 строк

Ассоциативная память для восстановления контекста по неполным данным.

```
struct SDM {
    hard_locations: Vec<[i8; D]>,    // D=256, ячейки-центры
    counters: Vec<Vec<i8>>,          // счётчики для каждой ячейки
    capacity: usize,                 // максимум ячеек (зависит от RAM)
}

impl SDM {
    fn read(address: &[f32]) → Vec<f32>;   // чтение по сходству
    fn write(address: &[f32], data: &[f32]); // запись
    fn cosine_near(query: &[f32], k: usize) → Vec<Vec<f32>>; // k похожих
}
```

**Оценка ёмкости:**
- 4 ГБ RAM → ~10⁹ ячеек (при D=256, 2 байта на ячейку)
- 32 ГБ RAM → ~10¹⁰ ячеек
- Поиск: O(log N) через KD-Tree

### 5.3. `src/tensor.rs` — Tensor Train (Оселедец–Тыртышников) ~500 строк

Сжатие DeepSeek-слоёв для работы на 8-16 ГБ.

```
struct TensorTrain {
    cores: Vec<Array3<f32>>,   // TT-ядра
    tt_rank: usize,            // ранг TT (10-30)
    original_shape: Vec<usize>, // форма исходного тензора
}

impl TensorTrain {
    fn compress(tensor: &ArrayD<f32>, rank: usize) → Self;  // сжатие
    fn decompress(&self) → ArrayD<f32>;                      // восстановление
    fn forward(&self, input: &[f32]) → Vec<f32>;            // прямой проход
    fn slice(&self, dim: usize, idx: usize) → Self;         // фрагмент для DTN
}
```

**Оценка:**
- Слой 500 МБ → TT-ранг 20 = 5 МБ (100x сжатие)
- Прямой проход: O(d × r² × n) вместо O(d × n²)
- Погрешность: < 1% при ранге 20

### 5.4. `src/pq.rs` — Product Quantization (Jégou) ~300 строк

Быстрый поиск по квантованным векторам.

```
struct ProductQuantizer {
    codebooks: Vec<Vec<Vec<f32>>>,  // M кодбуков по K центроидов
    codes: Vec<Vec<u8>>,            // кодовые слова (байты)
    m: usize,                       // число подпространств
    k: usize,                       // центроидов на подпространство
}

impl ProductQuantizer {
    fn train(vectors: &[Vec<f32>]) → Self;
    fn encode(vector: &[f32]) → Vec<u8>;
    fn search(query: &[f32], top_k: usize) → Vec<(usize, f32)>;
}
```

**Оценка:** D=768 → M=96, K=256 → 96 байт на вектор. Поиск 1M за 50 ms.

### 5.5. `src/federator.rs` — Кворум (BCM + АВО) ~400 строк

Сборка частичных ответов от N нод в единое решение.

```
struct Federator {
    method: FederationMethod,   // BCM | Avo | WeightedVote | Borda
    weights: Vec<f64>,          // вес каждой ноды (по истории)
    confidence_threshold: f64,  // порог принятия решения
}

impl Federator {
    fn aggregate_many(responses: &[NodeResponse]) → UnifiedResult;
    fn update_weights(&mut self, actual: &UnifiedResult);
}
```

Методы объединения:
- **BCM (Tresp):** P(y|x) = Π P_i(y|x)^{w_i} / Z — байесово произведение
- **АВО (Журавлёв):** Голосование с порогом: R(x) = Σ w_i·v_i(x) > θ
- **Weighted Vote:** Взвешенное среднее (быстрое)

### 5.6. `src/sensing.rs` — Compressed Sensing (Candès–Tao) ~300 строк

Редкая проекция знаний DeepSeek для DTN-передачи.

```
struct CompressedSensing {
    sensing_matrix: SparseMatrix,  // случайная разреженная матрица
    sparsity: f64,                 // предполагаемая редкость
}

impl CompressedSensing {
    fn project(data: &[f32]) → Vec<f32>;  // сжатие: n → O(k log n)
    fn recover(samples: &[f32]) → Vec<f32>; // восстановление через l1-оптимизацию
}
```

### 5.7. `src/tsetin.rs` — Автоматы Цетлина ~200 строк

Стратегия поведения ноды в случайной среде.

```
struct TsetlinAutomaton {
    states: Vec<u8>,          // состояния автомата (глубина памяти)
    actions: Vec<Action>,     // SEND_RESULTS | WAIT | RECEIVE | PROCESS_LOCAL
    rewards: Vec<f64>,        // накопленная награда за каждое действие
}

impl TsetlinAutomaton {
    fn decide(&self, context: &Environment) → Action;
    fn reward(&mut self, action: Action, signal: f64);
}
```

Автомат выбирает: отдать результаты / принять ситуации / работать локально.
Обучение без учителя — награда = количество успешно решённых ситуаций.

### 5.8. `src/mgua.rs` — МГУА (Ивахненко) ~400 строк

Самоорганизующаяся модель: растёт от фрагментов DeepSeek.

```
struct MGUA {
    layers: Vec<Vec<Polynomial>>,  // полиномиальные нейроны
    criterion: SelectionCriterion, // регулярность/смещение
    max_layers: usize,
}

impl MGUA {
    fn train(inputs: &[Vec<f64>], targets: &[f64]) → Self;
    fn predict(&self, input: &[f64]) → f64;
    fn partial_fit(&mut self, new_data: &[Vec<f64>]) → Self; // дообучение
}
```

Каждый фрагмент DeepSeek → новая точка для МГУА → модель уточняется.

### 5.9. `src/spec_decode.rs` — Speculative Decoding (Stern) ~200 строк

L0 генерирует гипотезу → DeepSeek верифицирует.

```
struct SpeculativeDecoder {
    draft_model: Box<dyn L0Model>,    // Марков/АВО (быстрый)
    num_draft_tokens: usize,          // K гипотетических токенов
}

impl SpeculativeDecoder {
    fn speculate(context: &[u32]) → Vec<u32>;      // L0 генерит K токенов
    fn verify(context: &[u32], draft: &[u32]) → Vec<u32>; // DeepSeek проверяет
}
```

### 5.10. `src/assoc_mem.rs` — Ассоциативная память (Amari) ~300 строк

Восстановление полного контекста из неполного.

```
struct AssociativeMemory {
    W: Array2<f32>,  // матрица ассоциативных связей
    n: usize,        // размерность
}

impl AssociativeMemory {
    fn train(pairs: &[(Vec<f32>, Vec<f32>)]) → Self;
    fn recall(partial: &[f32]) → Vec<f32>;  // восстановление
}
```

| Зависимость | Для | Лицензия |
|------------|-----|----------|
| `ndarray` | Tensor Train, SDM, МГУА | MIT |
| `ndarray-linalg` | SVD для TT | MIT |

| `blake3` | Хеши контекста | CC0 |
| `rayon` | Параллельные вычисления | MIT |

### 9.2. Внутренние

- `store.rs` — Redis/offline для хранения ситуаций
- `cargo.rs` — DTN-протокол для передачи фрагментов
- `node.rs` — метаданные о возможностях ноды (RAM, модули)
- `journal.rs` — бортовой журнал (обязательная запись каждого решения)

## 10. Риски

| Риск | Вероятность | Смягчение |
|------|------------|-----------|
| Tensor Train не сходится для данного слоя | Средняя | Запасной вариант: PQ вместо TT |
| SDM требует больше RAM, чем ожидалось | Высокая | Настраиваемое число ячеек |
| Кворум даёт < 70% при 2-3 нодах | Средняя | BCM-взвешивание с априорными вероятностями |
| Speculative Decoding ухудшает L0 | Низкая | Отключаемый feature gate |
| 2-3 одновременных разрыва связи | Низкая | OfflineQueue + CRDT merge |
